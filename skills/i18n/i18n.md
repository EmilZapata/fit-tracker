# Internationalization Patterns

i18next + react-i18next patterns for this project.

For API reference, see `docs/i18next.md`.

---

## Project Setup

### i18next Configuration

```tsx
// core/i18n/index.ts
import i18n from "i18next";
import { initReactI18next } from "react-i18next";

import en from "./locales/en";
import es from "./locales/es";

i18n.use(initReactI18next).init({
  resources: { en, es },
  lng: "en",
  fallbackLng: "en",
  compatibilityJSON: "v4", // required for React Native pluralization
  interpolation: {
    escapeValue: false, // React Native doesn't need HTML escaping
  },
  react: {
    useSuspense: false, // Suspense not well supported in React Native
  },
});

export default i18n;
```

Import `core/i18n` in the root layout (`app/_layout.tsx`) before any component renders:

```tsx
import "@/core/i18n";
```

### Language Persistence with MMKV

```tsx
// core/i18n/language-storage.ts
import { MMKV } from "react-native-mmkv";

const storage = new MMKV({ id: "app-settings" });
const LANGUAGE_KEY = "app-language";

export function getSavedLanguage(): string | undefined {
  return storage.getString(LANGUAGE_KEY);
}

export function saveLanguage(lng: string): void {
  storage.set(LANGUAGE_KEY, lng);
}
```

Use in i18n init:

```tsx
import { getSavedLanguage, saveLanguage } from "./language-storage";

const savedLng = getSavedLanguage();

i18n.use(initReactI18next).init({
  lng: savedLng || "en",
  // ...
});

i18n.on("languageChanged", saveLanguage);
```

## Translation File Structure

Organize by namespace, one file per namespace per language. Each namespace maps to a feature or domain.

```
core/i18n/
  index.ts              ← i18next init
  language-storage.ts   ← MMKV persistence
  locales/
    en/
      index.ts          ← re-exports all namespaces
      common.json       ← shared strings (buttons, labels, errors)
      workouts.json     ← workout feature strings
      exercises.json    ← exercise feature strings
      settings.json     ← settings screen strings
    es/
      index.ts
      common.json
      workouts.json
      exercises.json
      settings.json
```

### Locale Index File

```tsx
// core/i18n/locales/en/index.ts
import common from "./common.json";
import workouts from "./workouts.json";
import exercises from "./exercises.json";
import settings from "./settings.json";

export default { common, workouts, exercises, settings };
```

### Namespace Rules

- **`common`** — Default namespace. Shared UI: buttons ("save", "cancel", "delete"), generic labels ("loading", "error"), validation messages
- **Feature namespaces** — One per app domain. Keys scoped to that feature only
- Never duplicate keys across namespaces — if a string is reused, it belongs in `common`

## Translation Key Conventions

Use dot-nested keys with descriptive, hierarchical names:

```json
// common.json
{
  "button": {
    "save": "Save",
    "cancel": "Cancel",
    "delete": "Delete",
    "confirm": "Confirm"
  },
  "label": {
    "loading": "Loading...",
    "noResults": "No results found"
  },
  "error": {
    "generic": "Something went wrong",
    "network": "Check your internet connection"
  }
}
```

```json
// workouts.json
{
  "title": "Workouts",
  "create": {
    "title": "New Workout",
    "namePlaceholder": "Workout name"
  },
  "detail": {
    "sets": "{{count}} set",
    "sets_other": "{{count}} sets",
    "restTimer": "Rest: {{seconds}}s",
    "completed": "Workout completed!"
  },
  "empty": "No workouts yet. Start your first one!"
}
```

**Rules:**
- Use camelCase for keys
- Group by UI section: `create.title`, `detail.sets`, `list.empty`
- Pluralization uses `_other` suffix (v4 format): `"sets"` (one), `"sets_other"` (other)
- Interpolation variables in `{{double braces}}` — name them descriptively

## Usage in Components

### Basic Usage with useTranslation

```tsx
import { useTranslation } from "react-i18next";

export default function WorkoutsScreen() {
  const { t } = useTranslation("workouts");

  return (
    <View className="flex-1 p-4">
      <Text className="text-2xl font-bold">{t("title")}</Text>
    </View>
  );
}
```

### Multiple Namespaces

```tsx
const { t } = useTranslation(["workouts", "common"]);

// workouts namespace (first = default)
t("title");              // → "Workouts"

// common namespace (explicit prefix)
t("common:button.save"); // → "Save"
```

### Pluralization

```tsx
t("detail.sets", { count: 1 }); // → "1 set"
t("detail.sets", { count: 3 }); // → "3 sets"
```

### Interpolation

```tsx
t("detail.restTimer", { seconds: 90 }); // → "Rest: 90s"
```

## TypeScript Integration

Add type safety so `t()` autocompletes keys and catches typos.

```tsx
// core/i18n/i18next.d.ts
import "i18next";
import type en from "./locales/en";

declare module "i18next" {
  interface CustomTypeOptions {
    defaultNS: "common";
    resources: typeof en;
  }
}
```

This gives you autocomplete for all translation keys and compile-time errors for invalid keys.

## Language Switching

```tsx
import { useTranslation } from "react-i18next";

function LanguageSwitcher() {
  const { i18n } = useTranslation();

  const toggleLanguage = () => {
    const next = i18n.language === "en" ? "es" : "en";
    i18n.changeLanguage(next); // triggers MMKV save via languageChanged listener
  };

  return (
    <Pressable onPress={toggleLanguage}>
      <Text>{i18n.language.toUpperCase()}</Text>
    </Pressable>
  );
}
```

## Checklist for Adding a New Screen

1. Identify the namespace (existing feature or new one)
2. Add translation keys to all locale JSON files (start with `en`, then `es`)
3. Use `useTranslation("namespace")` in the component
4. Never hardcode user-facing strings — always use `t()`
5. If adding a new namespace, register it in the locale index files and update the TypeScript types
